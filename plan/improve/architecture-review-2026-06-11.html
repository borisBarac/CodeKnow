After

GraphSearcher

search(query, top_k) → HybridSearchResponse

internally: load graph, vector search, weighted BFS, chunk fetch, merge

Problem

hybrid_search() is a 100-line function importing from 7 modules across 4 sub-packages. It loads graphs, builds indices, runs vector search, runs BFS, fetches chunks, and merges results — all behind a single function call. Callers must supply correctly-wired ChromaStore + embeddings + graph path.

Solution

Deepen into a GraphSearcher module that owns the graph, reverse index, vector store, and embeddings as internal state. One search() method at the seam. multi_search folds in as a classmethod that discovers graphs and delegates.

Wins

  • leverage: one interface, N call sites (API, CLI, e2e)
  • locality: graph wiring bugs concentrate in one module
  • interface shrinks; implementation absorbs 5 shallow files
  • delete 2 pass-through modules (_utils.py, index.py)

Problem

Three files in the API package reach into PipelineConfig internals, import the private _env_path function, and manually construct ChromaStore + EmbeddingConfig + ChromaConfig from PipelineConfig attributes. ChromaDB wiring is duplicated between build and delete handlers.

Solution

Introduce a PipelineFacade in codeknow-lib that owns config, ChromaDB wiring, and repo_map. The API layer calls build(), delete(), and resolve_slug(). No private imports, no duplicated constructors.

Wins

  • seam between API and lib: API never sees ChromaConfig/EmbeddingConfig
  • locality: config + chroma wiring bugs concentrate in one module
  • leverage: 3 callers (app, models, middleware) share one module
  • delete private-symbol import (_env_path)

Deepen the extraction pipeline for testability

Strong in-process

codeknow/extract/ast.py (1076 lines — largest file in the repo)

codeknow/extract/detect.py (440 lines)

zero test coverage on both files

Before

collect_files() — file walk + graphignore
extract_python() — 300 lines, AST walk
extract_js() — 200 lines, AST walk
_resolve_cross_file_imports() — 140 lines
extract() — 150 lines, ID remapping + cache
extract_ast() — pipeline wrapper
no testable seam

After

Extractor.extract(repo_path) → ExtractionResult

internally: file walk, per-language AST, cross-file resolution, caching

testable through one seam

Problem

1516 lines of extraction logic with zero tests. The main extract() function mixes ID remapping, cross-file call resolution, cache orchestration, and language dispatch in one 150-line function. No seam exists for testing any part independently.

Solution

Deepen into an Extractor module with extract(repo_path) → ExtractionResult at the seam. Language-specific AST walkers and cross-file resolution become internal implementation. Tests exercise the full extraction through one interface against fixture repos.

Wins

  • interface is the test surface: test extraction end-to-end against fixture repos
  • locality: AST parsing bugs concentrate behind one seam
  • leverage: pipeline runner, e2e tests, cache layer all use one interface
  • internal restructure survives — tests describe behaviour, not implementation

After

DaemonLauncher

start(host, port) → pid

stop(timeout) → None

is_running() → bool

Both CLI and Client delegate to DaemonLauncher

Problem

Two independent paths to start the daemon: daemonocle via CLI (calls server.run_server which does which + subprocess.run) and DaemonManager via Client (builds a worker_command in endpoint.py). Different command construction, different lifecycle management. Understanding one gives zero insight into the other.

Solution

Unify into a single DaemonLauncher that both the CLI daemon command and Client.start_daemon() delegate to. One command construction path, one lifecycle model. Daemonocle stays as a thin CLI adapter.

Wins

  • locality: process lifecycle logic in one module, not split across 3
  • delete server.py (shallow: 18 lines, interface ≈ implementation)
  • eliminate fake_server reference in production endpoint.py
  • leverage: one module serves both CLI and programmatic paths

Problem

schemas.py defines Pydantic models with typed fields. validate.py reimplements the same validation using raw dict checks and duplicated constant lists (VALID_FILE_TYPES, VALID_CONFIDENCES). These two systems can drift: a new file type added to one but not the other silently breaks consistency.

Solution

Move all validation into Pydantic field validators inside schemas.py. validate.py's validate_extraction() becomes ExtractionResult.model_validate() with custom validators. Delete validate.py.

Wins

  • locality: one source of truth for valid values, not two
  • delete 140 lines of hand-rolled validation
  • leverage: Pydantic validators work at construction, JSON parse, and test time
Top recommendation

Deepen the extraction pipeline for testability

1516 lines with zero tests is the single highest-risk gap in the codebase. The extraction pipeline is the foundation everything else depends on — graph, chunks, embeddings, search. Without a testable seam, every change is a guess. Deepening here gives the highest leverage: tests at the interface protect all downstream modules, and the internal restructure (language walkers, cross-file resolution, caching) becomes safe to refactor. Start here because it makes every subsequent deepening safer.
